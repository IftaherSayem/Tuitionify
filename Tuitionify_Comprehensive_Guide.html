1 Project Overview

Tuitionify is a full-stack web application that serves as a tuition marketplace specifically designed for the International Islamic University Chittagong (IIUC) community and the broader Bangladeshi education system. It connects two groups of users:

  • Tutors — University students who offer private tutoring services
  • Seekers (Guardians/Students) — Parents or students looking to hire tutors

Core Value Proposition

Tuitionify eliminates the friction of finding tutors in Bangladesh by providing a trusted, verified, and filterable directory with privacy-aware contact sharing, review systems, and a structured application flow — all tailored to the Bangladeshi education system (BDT currency, local class levels like SSC/HSC, Chittagong areas/thanas).

2 System Architecture

4 Features & How They Work

4.1 Authentication System

How it works:

  1. Firebase handles identity — Users sign up/login via Email+Password or Google OAuth using the Firebase client SDK.
  2. Firebase sends a verification email on signup — users see a banner on their dashboard until verified.
  3. After Firebase login, a Mongo profile is created via POST /api/users/register. The server extracts uid, email, and name from the verified Firebase token.
  4. Every API request attaches the Firebase ID token via an Axios request interceptor (Authorization: Bearer <token>).
  5. The server verifies the token using Firebase Admin SDK before processing any protected request.

Supported auth flows:

  • Email/Password registration with email verification
  • Google OAuth sign-in (popup)
  • Password reset via email link
  • Email recovery (Firebase action handler)
  • Resend verification email from dashboard

4.2 Two-Role System (Tutor & Seeker)

How it works:

At registration, users choose their role: Tutor (university student offering tuition) or Seeker (guardian/student looking for a tutor). The role is stored in MongoDB and determines what features the user can access:

If a user signs in with Google but hasn't chosen a role yet, they are redirected to the /complete-profile page.

4.3 Tuition Posting & Application Flow

How it works:

Seeker posts tuition → Tutors browse & apply → Seeker reviews applicants → Accept / Reject
  • Posting: A seeker fills in title, class level, subjects, area, salary, days/week, mode (home/online), gender preference, and description. Stored in the Tuition collection with status open.
  • Applying: A tutor writes an optional message and submits. A unique compound index (tuition, tutor) prevents duplicate applications.
  • Managing applicants: The tuition owner sees all applicants with their profile info. They can accept or reject each one.
  • Auto-close: Accepting an applicant automatically sets the tuition status to closed.
  • Reopen: A seeker can toggle their tuition between open/closed from their dashboard.
  • Contact sharing: The tutor's phone number is only shown to the seeker after the application is accepted.

4.5 Privacy-Gated Contact System

How it works:

Tutor contact details (phone, email) are never publicly visible. A privacy-aware contact request system protects tutors:

  1. Public profile shows subjects, areas, rating, bio — but NOT phone or email
  2. Seeker sends a contact request with an optional introduction message
  3. Tutor reviews the request on their dashboard and approves or declines
  4. If approved, the seeker's tutor profile page reveals the phone and email
  5. Server enforces this — the publicTutor() function strips contact fields unless the viewer has an approved contact request

A unique compound index (tutor, seeker) ensures only one request per pair.

4.6 Review & Rating System

How it works:

  • Only seekers can leave reviews for tutors (1-5 star rating + optional comment)
  • A unique index (tutor, author) limits each seeker to one review per tutor
  • After each review, the server recomputes the tutor's average rating using a MongoDB aggregation pipeline ($avg, $sum)
  • The computed ratingAvg and ratingCount are denormalized onto the User document for fast listing queries
  • Reviews display on the tutor's public profile page

4.7 Reporting & Trust/Safety System

How it works:

  • Any logged-in user can report a tutor profile or tuition post via a modal dialog
  • Reports include a category (Fake profile, Spam, Inappropriate behavior, etc.) and optional details
  • Reports are stored in the Report collection with status: open, reviewed, or dismissed
  • Admins can review and update report statuses from the Admin panel

4.8 Admin Panel

How it works:

  • Admin access is determined by email whitelist in the server's ADMIN_EMAILS environment variable
  • The admin panel has two tabs:
    • Tutors Tab: Lists all tutors with verify/revoke badge buttons. Verified tutors get a badge and rank higher in search results.
    • Reports Tab: Lists all user/tuition reports with options to mark as reviewed or dismissed.
  • Client-side: ProtectedRoute checks profile.isAdmin before rendering the admin page
  • Server-side: requireAdmin middleware checks the user's email against the ADMIN_EMAILS list

4.9 Email Verification & Password Recovery

How it works:

  • On signup, Firebase sends a verification email automatically
  • A yellow banner appears on the dashboard if the email is unverified, with a "Resend" button
  • The /auth/action page handles Firebase's email action links (verify email, reset password, recover email)
  • The server syncs the emailVerified status from the Firebase token to MongoDB on each GET /users/me request
  • Password reset: user enters email on login page → Firebase sends reset link → /auth/action?mode=resetPassword lets them set a new password

4.10 Responsive UI & Design System

How it works:

  • Built with Tailwind CSS with a custom green brand palette (#1baf78 primary)
  • Custom component classes in index.css: .btn-primary, .btn-outline, .card, .badge, .chip, .input, .label
  • Mobile-first responsive design — sticky navbar with hamburger menu, grid layouts that stack on mobile
  • Code splitting via React's lazy() + Suspense for all page components
  • Loading states (spinner), empty states, and error states throughout the app
  • Toast notifications for all user actions (success, error, info)
  • Custom SVG logo with gradient for branding

4.11 Dark Mode

How it works:

  • Tailwind darkMode: 'class' — Dark mode is activated by toggling the dark class on the <html> element. A ThemeContext provider manages the state.
  • Persistent preference — The chosen theme is saved to localStorage('theme'). On first visit, falls back to the user's OS preference via prefers-color-scheme: dark.
  • Toggle button — A Sun/Moon icon button in the Navbar lets users switch themes with one click.
  • CSS component classes — All custom component classes (.btn-primary, .card, .input, .badge, .chip, .label) include dark: variants in index.css, so most components inherit dark styles automatically.
  • Page-level dark classes — Every page and component has been updated with inline dark: Tailwind classes for text, backgrounds, borders, and hover states.

4.12 Pagination

How it works:

  • Server-side pagination — Both GET /api/tuitions and GET /api/users/tutors accept page and limit query parameters (defaults: page=1, limit=12, max limit=50).
  • Response shape — Paginated endpoints return { data, page, totalPages, total } instead of a raw array.
  • Efficient queries — Uses Promise.all([find().skip().limit(), countDocuments()]) to fetch the page and total count in parallel.
  • Reusable Pagination component — Renders prev/next buttons and a page number window with ellipsis for large page counts.
  • Filter-aware — Page resets to 1 whenever filters or search queries change.

4.13 Search Bar

How it works:

  • Tuition search — The q query parameter triggers a MongoDB $or regex search across title, description, and subjects fields.
  • Tutor search — Expanded from name-only to $or across name, university, department, and subjects.
  • Debounced input — A 400ms debounce using a setTimeout ref pattern prevents excessive API calls while typing.
  • Search icon input — A styled search bar with a Search icon appears above the results grid on both Tuitions and Tutors pages.
  • Page reset — Changing the search query resets pagination to page 1.

4.14 Bookmark / Save Tuitions

How it works:

  • Toggle bookmark — Logged-in users can click a bookmark icon on any tuition card. POST /api/bookmarks/:tuitionId creates or deletes the bookmark (toggle pattern).
  • Batch check — When the tuitions page loads, GET /api/bookmarks/check?tuitionIds=... checks which tuitions on the current page are bookmarked, populating the bookmark icon state.
  • Saved tuitions dashboard — The Dashboard page includes a "Saved Tuitions" section that fetches GET /api/bookmarks and renders bookmarked tuition cards with a remove button.
  • Data model — A Bookmark model with a unique compound index on (user, tuition) prevents duplicates.
  • Click isolation — The bookmark button uses e.preventDefault() and e.stopPropagation() to prevent navigating to the tuition detail page when toggling a bookmark.

4.15 Profile Photo Upload

How it works:

  • Firebase Storage — Profile photos are uploaded to Firebase Storage at profile-photos/{uid}/{timestamp}_{filename}.
  • PhotoUpload component — A reusable component with a circular image preview and a camera icon overlay button. Clicking opens the native file picker.
  • Validation — Accepts only image/* files up to 5 MB in size.
  • Upload flow — Uses uploadBytes() to upload the file, then getDownloadURL() to get the public URL, which is saved to the user's photo field.
  • Replaces URL input — The old photo URL text input in the tutor profile form has been replaced with this visual uploader.

4.16 Edit / Delete Tuitions

How it works:

  • Edit tuition — Seekers can edit any of their posted tuitions via the /edit-tuition/:id page. The form pre-populates with the existing data and submits a PUT /api/tuitions/:id request.
  • Delete tuition — A delete button (with a confirmation modal) allows seekers to permanently remove a tuition post. Uses DELETE /api/tuitions/:id.
  • Cascade cleanup — Deleting a tuition also removes all related applications, bookmarks, and reports from the database.
  • Ownership verification — Both edit and delete endpoints verify that the logged-in user is the tuition's creator before allowing the operation.
  • Access points — Edit and Delete buttons appear in the Seeker Dashboard (My Posted Tuitions) and on the tuition detail page when the owner is viewing it.
  • Confirmation modal — A reusable ConfirmModal component prevents accidental deletions with a clear warning message.

4.17 Analytics Dashboard (Admin)

How it works:

  • Admin-only tab — A third "Analytics" tab in the admin panel displays key metrics, charts, and rankings.
  • Aggregated statistics — The server endpoint GET /api/admin/analytics returns total counts (users, tuitions, applications), weekly signup/posting trends, and top subjects/areas.
  • Charts with Recharts — Bar charts visualize signups and tuitions posted per week over the last 8 weeks.
  • Top rankings — The most popular subjects and areas are displayed as ranked lists with progress bars.
  • CSV Export — An "Export CSV" button generates a downloadable CSV file of all analytics data client-side.

4.18 Favorites / Saved Tutors

How it works:

  • TutorBookmark model — A new TutorBookmark collection with { user, tutor } fields and a unique compound index, mirroring the tuition bookmark pattern.
  • Toggle API — POST /api/tutor-bookmarks/:tutorId toggles the bookmark on/off. GET /check supports batch checking.
  • Heart icon on TutorCard — Each tutor card shows a heart icon that fills red when bookmarked. Uses e.stopPropagation() to prevent card navigation.
  • Dashboard section — A "Saved Tutors" section on the dashboard lists bookmarked tutors with the ability to remove them.

4.19 Tuition Recommendations

How it works:

  • Matching algorithm — GET /api/tuitions/recommended finds open tuitions matching the tutor's subjects, preferred areas, or class levels.
  • Overlap scoring — Results are scored by how many fields match (subjects worth 2 points, area and class level worth 1 each), sorted by score descending.
  • "Recommended for You" section — Appears at the top of the tutor dashboard when recommendations are available, displaying up to 6 matching tuitions.

4.20 Review Replies

How it works:

  • Reply fields on Review — The Review schema has reply (String) and replyAt (Date) fields added.
  • Tutor-only endpoint — PATCH /api/reviews/:id/reply allows only the reviewed tutor to post a single reply.
  • Display on profile — Replies appear below reviews with a styled left-border callout and "Tutor's reply" label.
  • Reply button — When viewing their own profile, tutors see a "Reply" button on unreplied reviews that reveals an inline textarea.

4.21 Application Tracking Timeline

How it works:

  • New timestamp fields — The Application model has viewedAt and decidedAt fields that record when the seeker first views and decides on the application.
  • Auto-view marking — When a seeker fetches applicants for their tuition, all unviewed applications are automatically marked as viewed.
  • Visual timeline — A 3-step dot timeline (Applied → Viewed → Decision) shows below each application, with timestamps and color-coded progress.
  • Both perspectives — The timeline is visible to both seekers (on the tuition detail page) and tutors (on their dashboard's "My Applications" section).

9 Security Analysis & Report

Overall Security Rating: GOOD for a University Project

The application demonstrates strong security awareness with Firebase-based authentication, server-side token verification, role-based access control, privacy-gated contact sharing, and proper CORS configuration. Several areas can be improved for production readiness.

9.1 Security Strengths

What the project does well:

  • Server-side token verification — Every protected endpoint verifies the Firebase ID token on the server using Firebase Admin SDK. Tokens cannot be forged.
  • Role-based access control — Both client-side (ProtectedRoute) and server-side (requireRole, requireAdmin) guards enforce role restrictions.
  • Privacy-gated contact info — Tutor phone/email are stripped from API responses unless the viewer has an approved contact request. The publicTutor() function and PRIVATE_FIELDS projection enforce this.
  • Ownership checks — Update/delete operations verify the requesting user owns the resource (e.g., "Not your tuition").
  • Unique constraints — Compound indexes prevent duplicate applications, reviews, and contact requests.
  • CORS configured — The server restricts the Access-Control-Allow-Origin to the client URL.
  • Environment variables for secrets — MongoDB URI, Firebase credentials, and admin emails are in .env (not hardcoded).
  • .env files excluded from git — The .gitignore properly excludes .env files; only .env.example templates are tracked.
  • Whitelisted field updates — PUT /users/me only accepts a specific list of editable fields, preventing privilege escalation (can't change role, isVerified, etc.).
  • Error handling — Central error handler catches and logs errors without leaking stack traces to clients.
  • Enum validation — Mongoose schemas use enums to constrain field values.

9.2 Security Findings

CRITICAL Real credentials in local .env files

Location: server/.env, client/.env

Issue: The local .env files contain real production credentials (MongoDB Atlas password, Firebase private key, Firebase API key). While these are NOT tracked in git (correctly excluded by .gitignore), if the repository is ever shared, copied, or backed up with these files included, all credentials would be exposed.

Impact: An attacker with the MongoDB URI could read/write/delete all database records. The Firebase private key could be used to impersonate any user.

Recommendation:

  • Rotate all credentials immediately if you suspect they've been shared
  • Never copy .env files when sharing the project
  • Use environment-specific credential management (Vercel env vars for production)
  • Consider IP-whitelisting the MongoDB Atlas cluster
MEDIUM Regex injection via name search

Location: server/routes/users.js:114

Issue: The tutor search uses { $regex: q, $options: 'i' } where q comes directly from user input. Malicious regex patterns (e.g., ((((((((((a])*)*)*)*)*)*)*)*)*)*) can cause catastrophic backtracking (ReDoS), hanging the server.

Recommendation: Escape the user input before using it in a regex, or use MongoDB's $text index for full-text search.

// Fix: escape regex special characters
const escaped = q.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
filter.name = { $regex: escaped, $options: 'i' };
MEDIUM Mass assignment on tuition creation

Location: server/routes/tuitions.js:51

Issue: Tuition.create({ ...req.body, createdBy: req.dbUser._id }) spreads the entire request body into the document. While Mongoose schema validation will reject unknown fields, a user could potentially set status: 'closed' or override createdBy (though the explicit createdBy overrides it).

Recommendation: Destructure only the expected fields from req.body, similar to the profile update endpoint.

MEDIUM No rate limiting

Location: server/index.js

Issue: The API has no rate limiting middleware. This makes it vulnerable to brute-force attacks, scraping, and abuse of write endpoints (spamming reports, reviews, applications).

Recommendation: Add express-rate-limit middleware, especially on auth-related and write endpoints.

import rateLimit from 'express-rate-limit';
const limiter = rateLimit({ windowMs: 15 * 60 * 1000, max: 100 });
app.use('/api', limiter);
MEDIUM No input sanitization for stored text

Location: All routes accepting user text (titles, descriptions, bios, messages, reviews)

Issue: User-submitted text (tuition titles, descriptions, bio, review comments, application messages) is stored as-is in MongoDB and rendered in React. While React auto-escapes JSX content (preventing XSS), the data could be problematic if ever rendered in non-React contexts (email notifications, server-rendered pages, admin exports).

Recommendation: Add a sanitization layer (e.g., xss or sanitize-html) for stored user text as a defense-in-depth measure.

LOW No request body size limit

Location: server/index.js:22

Issue: express.json() defaults to a 100KB body limit, which is reasonable. However, explicitly setting it and adding validation for field lengths (e.g., max bio length, max description length) would be more robust.

Recommendation: Add express.json({ limit: '100kb' }) explicitly and add Mongoose maxlength validators to string fields.

LOW Missing security headers

Location: server/index.js

Issue: The API doesn't set security headers like X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security, etc.

Recommendation: Add the helmet middleware for automatic security headers.

import helmet from 'helmet';
app.use(helmet());
LOW No pagination on list endpoints

Location: All GET list endpoints

Issue: Tutor/tuition list endpoints return all matching results without pagination. With a large dataset, this could lead to slow responses and high memory usage.

Status: FIXED — Server-side pagination with page/limit query params has been added to tuition and tutor listing endpoints. See Section 4.12.

10 Deployment Guide

10.1 Architecture

Tuitionify deploys as two separate Vercel projects from a single repository:

ProjectRoot DirectoryFrameworkURL Pattern
API (server)server/Other (Node.js)api.tuitionify.vercel.app
Clientclient/Vitetuitionify.vercel.app

10.2 Server Deployment

  1. Import the repository into Vercel
  2. Set Root Directory to server
  3. Set Framework Preset to Other
  4. Add all environment variables from server/.env.example
  5. Set CLIENT_URL to the client's deployed URL
  6. The vercel.json in server/ configures the Node.js serverless function and rewrites all routes to index.js

10.3 Client Deployment

  1. Import the repository again (or configure a second project)
  2. Set Root Directory to client
  3. Set Framework Preset to Vite
  4. Add all VITE_* environment variables from client/.env.example
  5. Set VITE_API_URL to https://<your-api-url>/api
  6. The vercel.json in client/ rewrites all routes to index.html for SPA routing

10.4 Firebase Configuration

  • Add the client's deployed domain to Firebase Console → Authentication → Settings → Authorized domains
  • If using custom domain, add it as well
  • Ensure Email/Password and Google sign-in providers are enabled

10.5 MongoDB Atlas Configuration

  • Allow Vercel's IP ranges in Network Access (or 0.0.0.0/0 for serverless)
  • The server uses connection caching for serverless (Vercel reuses the process between requests)

10.6 Local Development

# Terminal 1 - Backend
cd server && npm install && npm run dev    # http://localhost:5000

# Terminal 2 - Frontend
cd client && npm install && npm run dev    # http://localhost:5173

# Optional: Seed demo data
